## Context

The current `VideoLessonDetailScreen` in the `courses` package hardcodes four subtabs: Notes, Transcript, Ask Doubt, and AI Support. This creates a confusing user experience when a video does not support these features, as users see empty or irrelevant tabs. The API (v2.4 `contents/` endpoint) now provides metadata flags and URLs (e.g., `enable_transcript`, `video_subtitle`, `is_ai_enabled`, `ai_notes_url`) to indicate which features are available and where to fetch the corresponding content. We need to implement dynamic tab rendering and proper data parsing for these sub-features.

## Goals / Non-Goals

**Goals:**
- Dynamically render the subtabs in `VideoLessonDetailScreen` based on `Lesson` properties.
- Map the new API fields to `LessonDto` and `Lesson` domain models.
- Implement the "Transcript" tab to fetch, parse, and render WebVTT (`.vtt`) files when `job_status` is `COMPLETED`. Show "Transcription in progress" otherwise.
- Implement the "Notes" tab to fetch and render Markdown (`.md`) files from `ai_notes_url` when `is_ai_enabled` is true.

**Non-Goals:**
- Re-architecting the overarching `VideoLessonDetailScreen` layout (header, player, etc. remain the same).
- Modifying the AI Support backend or Ask Doubt functionality beyond toggling their visibility.
- Implementing offline caching for transcripts and notes (out of scope for this initial integration).

## Decisions

1. **Model Updates**:
   - Add `bool enableTranscript`, `String? videoSubtitleUrl`, `bool isAiEnabled`, and `String? aiNotesUrl` to `LessonDto` and `Lesson`.
   - `videoSubtitleUrl` is resolved at parse time from the `video_subtitle` JSON object: it is set to the subtitle URL only when `job_status` is `"COMPLETED"`, and `null` otherwise. This avoids storing the raw map and bakes the completion check into the model itself.
   - Update `_parseVideoLesson` and `_parseBase` in `LessonDto` to safely extract these fields from the JSON payload.
   - Update the `isComplete` getter in `Lesson` so that `LessonType.video` is only considered complete once `isDetailFetched` is true. This ensures `LessonDetailOrchestrator` displays a loading indicator until the full metadata is ready, avoiding the layout jump where only one tab is initially shown before the others appear.

2. **Dynamic TabController**:
   - In `_VideoLessonDetailScreenState`, instead of hardcoding `TabController(length: 4)`, we will dynamically build the list of active tabs (`List<Widget>`) based on the `widget.lesson` properties.
   - We will recreate the `TabController` if the number of active tabs changes, using `didUpdateWidget` to manage updates safely.

3. **Clean Architecture & Data Fetching (VideoSubtabsRepository)**:
   - Introduce a dedicated `VideoSubtabsRepository` at `packages/courses/lib/repositories/video_subtabs_repository.dart` that accepts `Dio` and contains the low-level HTTP calls and parsing logic.
   - Declare a singleton repository provider: `videoSubtabsRepositoryProvider` generated by `@riverpod`.
   - Declare two family state providers:
     - `fetchNotesMarkdownProvider(String url)`: Calls `repository.fetchNotesMarkdown(url)`.
     - `fetchTranscriptProvider(String url)`: Calls `repository.fetchTranscriptCues(url)`.

4. **Transcript Tab UI & WebVTT Parsing**:
   - The UI will check the status. If completed, it will watch `fetchTranscriptProvider(subtitleUrl)`.
   - The repository handles fetching the VTT content and uses a background isolate via `compute(VttParser.parse, data)` to parse VTT cues off the main thread, returning a cleanly structured list of cues.
   - If not completed, the UI shows a friendly "Transcription in progress" empty state.
   - During the `loading` state of `fetchTranscriptProvider`, a high-fidelity `Skeletonizer` loading state (mocking transcript cues) will be displayed. This ensures structural height is maintained, preventing nested footers from floating up.

5. **Notes Tab UI**:
   - If `isAiEnabled` is true and `aiNotesUrl` is not null/empty, show the Notes Tab.
   - The Notes Tab UI watches `fetchNotesMarkdownProvider(notesUrl)`.
   - The UI renders the fetched raw markdown content using the `flutter_markdown` package (`MarkdownBody`) for elegant presentation with clean scroll boundaries.
   - During the `loading` state, a `Skeletonizer` loading state (mocking text paragraphs) will be displayed to provide a premium UX and maintain layout height.

6. **Sliver Scrolling Architecture**:
   - The original implementation wrapped tab contents inside a `SingleChildScrollView` with `shrinkWrap: true` on child list views. This inadvertently disabled lazy loading, causing massive UI freezes when rendering large transcripts on the main thread.
   - We will utilize a Slivers-based layout in `VideoLessonViewer` (`CustomScrollView`) and update `TranscriptsTab` to output a `SliverList`. This restores lazy loading, keeps the main thread unblocked during render, and supports a sticky `SliverToBoxAdapter` for the footer.

## Risks / Trade-offs

- **Risk**: Dynamic `TabController` recreation might cause UI stutter if not handled correctly.
  - *Mitigation*: Only recreate the controller when the length of active tabs actually changes during `didUpdateWidget`.
- **Risk**: Parsing large `.vtt` files on the main thread might drop frames and cause interface jank.
  - *Mitigation*: The repository delegates WebVTT parser logic to a background Dart Isolate via `compute()`. This keeps the main Flutter thread pristine and fully responsive.
- **Risk**: Missing markdown or VTT fetching error handling.
  - *Mitigation*: Wrap fetching operations in robust `AsyncValue` patterns in the UI, ensuring clear error states with dedicated "Retry" action triggers.
