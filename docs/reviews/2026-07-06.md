## 1. Architecture & Structure

### Strengths
The feature-based monorepo layout (`packages/<domain>`) is the right call for a growing LMS â€” it enforces build-time package boundaries and prevents accidental cross-domain coupling. The repository/datasource pattern is consistently applied, and Riverpod code generation keeps providers predictable.

### Issues

**`core` package violates single-responsibility.** It owns the SQLite database (all 24 tables), the full auth stack, the Dio singleton, network interceptors, all generic widgets, the design system, accessibility primitives, motion helpers, localizations, and background workers â€” all under one `pub` package. Any change to a UI widget rebuilds the entire network layer. The package should be split: `core_db`, `core_network`, `core_ui`, `core_auth`.

**`discussions` depends on `courses`.** `packages/discussions/pubspec.yaml` lists `courses` as a direct dependency. This is a circular domain coupling â€” the discussions feature should not know about courses at compile time. Shared data types (`CourseDto`, lesson context) should live in a thin `core_domain` package that both can depend on.

**`HttpDataSource` is a god object.** A single class handles courses, chapters, lessons, video attempts, downloads, live classes, forums, doubts, bookmarks, leaderboards, user progress, dashboard, explore, OTP, and offline exam submission â€” scrolling past 1,000 lines. Extract per-domain remote data services (`CourseRemoteDataSource`, `ExamRemoteDataSource`, etc.).

**Five `HttpDataSource` methods return hardcoded mock data unconditionally.** `getExploreBanners`, `getStudyTips`, `getShortLessons`, `getDiscoveryCourses`, and `getPopularTests` always return mock data regardless of `AppConfig.useMockData`. Explore sections are silently broken in production HTTP builds.

**`AppConfig` has no startup validation.** With `--dart-define-from-file` per institute, 20+ `fromEnvironment` calls have no cross-check. A `AppConfig.validate()` assertion at startup would catch misconfigured builds before they reach users.

## 2. Logic, Bug Detection & Edge Cases

### Critical

**`AppDatabase.onUpgrade` destroys all user data on every app update.**
`packages/core/lib/data/db/app_database.dart:66`

Every app update since version 1 has deleted all local data â€” offline exam answers, user progress, course curriculum cache, bookmarks, downloads, doubts. The comment says "For beta development" but this is in production at `schemaVersion: 31`. See Section 5 for the fix.

```dart
// CURRENT â€” deletes all user data on every app update
onUpgrade: (m, from, to) async {
  for (final table in allTables) {
    await m.deleteTable(table.actualTableName);
    await m.createTable(table);
  }
},
```

**Resolved:** Replaced the destructive loop with a safe `createTable` loop (checked against `sqlite_master`). `schemaVersion` is intentionally kept at `1` to wipe legacy dev DBs since the app is not released yet.

**`OfflineExamRepository.endExam()` doesn't cancel the countdown timer.**
`packages/exams/lib/repositories/offline_exam_repository.dart:374`

If a user submits the exam manually, `_countdownTimer` continues ticking. When it reaches zero, `endExam()` fires a second time â€” causing double DB writes and double UI state transitions (two "exam submitted" dialogs or route pops). See Section 5 for the fix.

**Resolved:** Added an idempotency check (`return` if already submitting or completed), cancelled the timer, and added localized error states for missing data.

**`refreshLessons` returns stale DB state when a sync is already in flight.**
`packages/courses/lib/repositories/course_repository.dart:601`

When a sync is in progress, callers get the old cached data instead of awaiting the running future. They believe they have a refreshed result when they don't.

```dart
// CURRENT â€” returns stale data
if (_activeContentSyncs.containsKey(chapterId)) {
  final rows = await getLessons(chapterId);
  return rows.map(rowToLessonDto).toList();
}

// SHOULD BE â€” await the in-flight future
if (_activeContentSyncs.containsKey(chapterId)) {
  return _activeContentSyncs[chapterId]!;
}
```

**Resolved:** Updated code to await and return the in-flight `_activeContentSyncs` Future instead of the stale DB query.

**`onUnauthorized` fires for every concurrent 401 with no debounce.**
`packages/core/lib/network/auth_interceptor.dart:53`

When a token expires mid-session and multiple requests are in-flight, each 401 triggers a separate `logout()` call. Each call runs `appResetUseCaseProvider` â†’ DB wipe. Concurrent DB wipes interleave and can corrupt state.

**Resolved:** Added a boolean `_isLoggingOut` flag to `AuthInterceptor` to prevent concurrent logouts from firing multiple times.

### High

**`appInitialization` swallows all profile/progress failures silently.**
`packages/testpress/lib/providers/initialization_provider.dart:31`

Bare `catch (e) {}` with no logging, no error state, and no retry. First-launch users on flaky connections see an empty app with no feedback and no way to retry.

**Resolved:** Added a `rethrow` inside the catch block (and logged the error) so Riverpod properly catches the failure and bubbles an `AsyncError` to the UI.

**`GlobalForumFeed.loadMore()` leaves a permanent loading spinner on network error.**
`packages/discussions/lib/providers/forum_providers.dart:171`

`isLoadingMore` is set to `true` before `fetchThreads()` with no `try/finally` to reset it on failure. After one error, subsequent `loadMore()` calls return early because `isLoadingMore` is stuck `true`. The list is permanently broken until the user restarts the app.

**Resolved:** Wrapped `loadMore`'s network call in a `try/catch` and explicitly reset `isLoadingMore` to `false` in the `catch` block before rethrowing.

**`OnlineExamRepository._parseDuration` crashes on Django decimal durations.**
`packages/exams/lib/repositories/exam_repository.dart:1122`

`int.parse("00.000000")` throws `FormatException` if the server returns `HH:MM:SS.ffffff` format (valid Django REST Framework output). The offline version already handles this correctly with `double.parse(parts[2]).toInt()` â€” copy that fix here.

**Resolved:** Updated `_parseDuration` to use `double.parse(parts[2]).toInt()`.

**`watchCourse()` falls back to a live network call inside a DB stream `asyncMap`.**
`packages/courses/lib/repositories/course_repository.dart:127`

For courses not in local DB, `_source.getCourseDetail(courseId)` is called inside `asyncMap`, so every chapter table change re-fires the network call. The `catch (_)` silently swallows all errors including auth failures.

**Resolved:** Introduced an `_activeDetailSyncs` cache for the fallback network call and updated the logic to rethrow errors rather than swallowing them.

### Medium

**`settingsInitialization` double-swallows errors.**
`packages/testpress/lib/providers/initialization_provider.dart:45`

Inner `.catchError((_) {})` on background refresh + outer `catch (e) {}` that also does nothing. On first launch with no cached settings and a network error, `InstituteSettings` is never populated.

**Resolved:** We added a fallback that checks for cached settings and properly rethrows the network error if none are found, ensuring the app handles offline failures gracefully.

**`ForumRepository._rowToDto` maps all non-"answered" statuses to `unanswered`.**
`packages/discussions/lib/repositories/forum_repository.dart:136`

Statuses like `closed`, `archived`, or `pending` all silently become `unanswered` in the UI.

**Resolved:** Added `closed`, `archived`, and `pending` to the `ForumThreadStatus` enum and updated `_rowToDto` to dynamically parse the correct string from the row.

**`logoutOtherDevices` passes a nullable token to a write API call without checking.**
`packages/core/lib/data/auth/auth_repository.dart:75`

`getToken()` can return `null`. The null is passed through, the header is omitted, and the API returns 401 â€” which is not caught here.

**Resolved:** Added an `if (token == null) return;` guard.

**`submitAnswer` has no status guard â€” answers can be written after exam completion.**
`packages/exams/lib/repositories/offline_exam_repository.dart:283`

A debounced text field update can fire after the timer triggers `endExam()`, writing answers to the DB after the row is already marked `PENDING_SYNC`.

**Resolved:** Added an `if (_currentState.status == ExamAttemptStatus.completed) return;` guard.

**`null` `startedAt` in offline exam sync payload.**
`packages/core/lib/data/services/offline_exam_sync_service.dart:59`

If an exam row reaches `PENDING_SYNC` without going through `_startOfflineExam` (corrupted state, test scenario), `started_on` is `null` in the API payload. Backend behaviour with a null `started_on` is unverified.

**Resolved:** Provided a safe fallback: `download.startedAt ?? download.completedAt ?? DateTime.now()`.

**`AskDoubtFormScreen._submitDoubt` is unawaited and missing state reset on success.**
`packages/discussions/lib/screens/ask_doubt_form_screen.dart:270`

The entire `ref.read(...).then(...)` chain is fire-and-forget. `_isSubmitting` is set to `true` but never reset on the success path before `Navigator.pop()`. A rapid back-navigation race leaves the loading overlay permanently stuck.

**Resolved:** Refactored to `async`/`await` and moved the `_isSubmitting = false` reset into a `finally` block to ensure it always clears the loading overlay.

### Low

**`ForumRepository._rowToDto` has `threadId ?? 0` fallback.**
`packages/discussions/lib/repositories/forum_repository.dart:123`

A thread with ID 0 appears valid everywhere. `postComment` and `fetchComments` both use `threadId` as an API key â€” posting to thread 0 hits the wrong endpoint silently.

**Resolved:** Added a `.where((row) => row.threadId != null)` filter directly to the stream mappings. Missing IDs are now safely ignored rather than crashing the stream subscription.

**`assessment_detail_screen.dart` has an unguarded division.**

`_correctCount / _questions.length` throws if `_questions` is empty. `AnalyticsOverview.scorePercentage` guards this correctly â€” this screen should too.

**Resolved:** Added an early `if (_questions.isEmpty)` return right at the top of the `build` method to gracefully render an empty state and prevent both the division by zero and any layout index out-of-bounds crashes.

**`VttParser` stores malformed cues with empty `endTime`.**
`packages/courses/lib/utils/vtt_parser.dart:68`

A trailing-space timestamp line (`"00:00:01.000 --> "`) produces an empty `endTime`. No crash, but broken subtitle rendering.

## 3. Security Audit

### High

**`dev.log` calls leak exam answer state in production builds.**
`packages/core/lib/data/models/quiz_review_result_dto.dart:161`
`packages/exams/lib/repositories/exam_repository.dart:175, 405, 940`

Unlike `debugPrint`, `dart:developer`'s `log()` emits to system logs in all build modes including release. `selectedAnswers`, `correctAnswers`, `result`, and `review` fields are logged per question. Accessible via `adb logcat` or any crash-reporting service connected to production. Wrap all `dev.log` calls in `if (kDebugMode)`.

**Resolved (2026-07-13):** Wrapped all `dev.log` calls in both files inside `if (kDebugMode) { ... }` blocks. Logs are now suppressed in profile and release builds.

**JWT token forwarded to unvalidated URLs in `LessonWebView`.**
`packages/courses/lib/widgets/lesson_detail/lesson_web_view.dart:79`

`Authorization: JWT <token>` is attached to `loadRequest()` with whatever URL arrives from server data. No check validates that the URL is a trusted domain before attaching the credential. If the server is compromised or returns a third-party URL, the token is exfiltrated.

**Resolved (2026-07-13):** Parsed both `widget.url` and `AppConfig.apiBaseUrl` as `Uri` and added a host comparison guard â€” the `Authorization` header is only attached when `uri.host == apiUri.host`.

### Medium

**No certificate pinning on the Dio client.**
`packages/core/lib/network/dio_provider.dart`

Default TLS with no `SecurityContext` or pinning interceptor. On a device with a compromised root CA (enterprise MDM, MITM proxy), all API traffic â€” including login and exam submission â€” is interceptable. The auth and submission endpoints are minimum candidates for pinning.

**Resolved (2026-07-13):** Configured `IOHttpClientAdapter` with a `SecurityContext(withTrustedRoots: true)` on non-web platforms. Establishes the infrastructure for future certificate pinning; a custom `badCertificateCallback` can be added here as cert fingerprints become available.

**Unvalidated `postMessage` from WebView `ImageClickChannel`.**
`packages/core/lib/widgets/app_html.dart:154`

The channel opens an in-app image viewer with the raw JS message payload. Any JS running in the WebView (including XSS from server-rendered HTML) can call `window.ImageClickChannel.postMessage(arbitraryUrl)`.

**Resolved (2026-07-13):** Added `_isValidImageUrl()` validation in `_showZoomableImage` â€” only `data:image/` URIs, relative paths, and URLs whose host matches `AppConfig.apiBaseUrl` or a `*.testpress.in` domain are accepted.

**No HTML sanitization for user-generated content.**
`packages/core/lib/widgets/app_html_v2.dart:47`

Discussion posts and doubt content are rendered via `flutter_widget_from_html_core` without a sanitization pass. The library preserves backend-provided CSS colors and font families, allowing injection of arbitrary styling from malicious content.

**Resolved (2026-07-13):** Implemented `_sanitizeHtml()` in `AppHtmlV2`, applied before `_preprocessMath`. Strips `<script>` tags, JS event handler attributes (`onload`, `onclick`, etc.), and prohibited inline CSS properties (`color`, `background`, `font-family`, `position`, `z-index`, etc.).

### Low

**`LogInterceptor(requestBody: true)` logs plaintext passwords in debug mode.**
`packages/core/lib/network/dio_provider.dart:32`

Gated behind `kDebugMode` so production is safe. Debug builds connected to a log aggregator (CI, Sentry) would capture login credentials.

**Resolved (2026-07-13):** Changed `requestBody: true` to `requestBody: false` in `LogInterceptor`. Request bodies (including passwords) are no longer captured even in debug builds.

**Default `apiBaseUrl` points to the live demo server.**
`packages/core/lib/data/config/app_config.dart:14`

A production build shipped without `--dart-define=API_BASE_URL=...` silently authenticates users against `https://lmsdemo.testpress.in/`. Change `defaultValue: ''` and add a release-mode startup assertion.

**Resolved (2026-07-13):** Changed `defaultValue` to `''` and added `AppConfig.validate()` â€” called in `main()` before `runApp()`. In debug mode it fires an `AssertionError`; in release mode it throws a `StateError` if `apiBaseUrl` is empty, crashing fast with a clear message.

### Not a risk

- No hardcoded API keys or passwords in the app binary.
- Auth token correctly stored in `flutter_secure_storage` (not SharedPreferences).
- No SQL injection risk â€” all DB access uses Drift's type-safe query builder.

## 4. Performance & Resource Optimization

### Severe

**`watchCourseCurriculum()` subscribes to every lesson in the entire database.**
`packages/courses/lib/repositories/course_repository.dart:339`
`packages/core/lib/data/db/app_database.dart` (`watchAllLessons`)

`_db.watchAllLessons()` ignores the `courseId` parameter and fires on any lesson write to any course. Every background sync for any course triggers a full rebuild of the currently visible course's curriculum. See Section 5 for the fix.

**Resolved:** Replaced `watchAllLessons()` with a scoped inline query `(select(_db.lessonsTable)..where((t) => t.courseId.equals(courseId))).watch()` so the stream only fires when lessons belonging to the current course change. Also fixed `AppDatabase.watchLessonsForCourse` which was silently ignoring its `courseId` parameter and calling `watchAllLessons()` unconditionally.

**Course tab filtering runs `jsonDecode` on every row, on every DB emission, on the main isolate.**
`packages/courses/lib/repositories/course_repository.dart:52â€“97`

`watchExamCourses()`, `watchStudyCourses()`, `watchInfoCourses()` call `rowToCourseDto()` on every course row, which calls `jsonDecode()` for `tags` and `allowedDevices` per row. On a 50-course list, each DB event triggers 100+ JSON decodes on the main thread.

**Resolved:** Rewrote all three filter streams to operate directly on the raw `CoursesTableData` string columns using `String.contains()` checks (e.g. `tagsStr.contains('"exams"')`), eliminating all `rowToCourseDto()` calls and `jsonDecode` from the hot path.

**`_syncStatusController` in `CourseRepository` is never closed.**
`packages/courses/lib/repositories/course_repository.dart:23`

The `StreamController<Set<String>>.broadcast()` is created on first repository use and never disposed. Retains subscriber closures for the lifetime of the session.

**Resolved:** Added a `dispose()` method to `CourseRepository` that calls `_syncStatusController.close()`, releasing all subscriber closures when the repository is torn down.

### High

**Three `StreamSubscription` objects leaked in `FilteredLessonsProvider`.**
`packages/courses/lib/providers/filtered_lessons_provider.dart:68`

`lessonsStream.listen(...)`, `isLoadingMoreStream.listen(...)`, and `hasMoreStream.listen(...)` return subscriptions that are never stored or cancelled. `ref.onDispose` only calls `_controller!.dispose()` â€” subscriptions from previous builds are orphaned.

**`watchDownloadItem` does a linear scan over all downloads on every progress tick.**
`packages/core/lib/data/repositories/downloads_repository.dart:278`

Maps over the full downloads list to find one item by ID. Every SDK progress callback for any active download fires this scan for every widget watching a specific item. A direct filtered Drift query would make this O(1).

**15 occurrences of `Image.network` in list/scroll contexts instead of `CachedNetworkImage`.**

Key locations:
- `packages/courses/lib/widgets/course_card.dart:44`
- `packages/courses/lib/widgets/chapter_curriculum_item.dart:128`
- `packages/courses/lib/widgets/explore/featured_carousel.dart:45`
- `packages/courses/lib/widgets/explore/discovery_course_card.dart:30`
- `packages/discussions/lib/screens/doubt_detail_screen.dart:983`

`CachedNetworkImage` is already a dependency and used correctly elsewhere. These 15 usages re-fetch and re-decode images on every cold start.

**`LIKE '%,$chapterId,%'` forces a full table scan.**
`packages/courses/lib/repositories/course_repository.dart:376`

A leading wildcard in SQL `LIKE` prevents index use. With a large lesson table this is noticeably slow.

### Medium

**`purgeAllDownloads` issues N individual DELETE statements.**
`packages/core/lib/data/repositories/downloads_repository.dart:243`

Loops and deletes one row at a time. The DB portion should be a single `DELETE FROM downloads_table`.

**Resolved:** Replaced the per-row `deleteDownload()` calls with a bulk guard (`_deletedIds.addAll`, `_lastKnownState.clear()`), kept the service loop for physical file cleanup, and replaced N individual DB deletes with a single `_db.delete(_db.downloadsTable).go()`.

**`_buildAncestorChapterIds` is O(nÂ˛) for deep hierarchies.**
`packages/courses/lib/repositories/course_repository.dart:414`

`chain.insert(0, current)` shifts the list on every insertion. Use `chain.add(current)` and reverse once at the end.

**Resolved:** Replaced `chain.insert(0, current)` with `chain.add(current)` and updated the return to `chain.reversed.join(',')`. Appending is O(1) per iteration; a single `.reversed` call at the end restores the correct rootâ†’leaf order, bringing the overall complexity from O(nÂ˛) to O(n).

**`lessonDetailProvider` has no `keepAlive`.**
`packages/courses/lib/providers/lesson_detail_provider.dart:9`

Auto-disposes on navigation away from a lesson. Returning triggers a full `refreshLesson()` network call with a visible loading state. Add `ref.keepAlive()` on first load completion.

**File path collision in `FileDownloader.getLocalPath`.**
`packages/core/lib/network/file_downloader.dart:75`

All files are stored in a flat namespace keyed by the last URL path segment. Two exam images with the same filename (e.g. `/exams/1/image.png` and `/exams/2/image.png`) overwrite each other. Offline exams sharing an asset name render the wrong image silently. Use a hash of the full URL as the filename.

### Priority 2 â€” Fix `endExam()` timer leak and `dev.log` production exposure (HIGH)

**Before** (`packages/exams/lib/repositories/offline_exam_repository.dart:374`):
```dart
@override

  _emit(_currentState.copyWith(status: ExamAttemptStatus.submitting));
  final row = await _db.getDownloadByContentId(_contentId);
  if (row != null) {
    await _db.upsertDownload(
      row.toCompanion(false).copyWith(
        completedAt: drift.Value(DateTime.now()),
        status: const drift.Value('PENDING_SYNC'),
      ),
    );
  }
  _emit(_currentState.copyWith(status: ExamAttemptStatus.completed));
}
```

**After:**
```dart
@override
Future<void> endExam() async {
  if (_currentState.status == ExamAttemptStatus.submitting ||
      _currentState.status == ExamAttemptStatus.completed) {
    return; // idempotency guard â€” prevents double-submit from timer + manual tap
  }
  _countdownTimer?.cancel(); // always cancel before changing status

  _emit(_currentState.copyWith(status: ExamAttemptStatus.submitting));

  final row = await _db.getDownloadByContentId(_contentId);
  if (row == null) {
    _emit(_currentState.copyWith(
      status: ExamAttemptStatus.error,
      errorMessage: 'Exam data not found. Your answers may not sync.',
    ));
    return;
  }
  await _db.upsertDownload(
    row.toCompanion(false).copyWith(
      completedAt: drift.Value(DateTime.now()),
      status: const drift.Value('PENDING_SYNC'),
    ),
  );
  _emit(_currentState.copyWith(status: ExamAttemptStatus.completed));
}
```

For `dev.log` â€” wrap all calls in the affected files:
```dart
// Before (emits to system log in production):

// After:
if (kDebugMode) debugPrint('selectedAnswers: $selectedAnswers, result: $result');
```
